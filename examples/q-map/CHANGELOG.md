# Changelog

All notable changes to `examples/q-map` should be documented in this file.

## Unreleased

### Refactored
- **Split `cloud-tools.tsx` (3363 LOC) into `cloud/` submodules:** 10 focused modules (constants, cache, providers, schemas, dataset-identity, catalog, query-core, tool-factories, components, index barrel). Original file is now a thin re-export barrel for backward compatibility. Updated `audit-tool-coverage.mjs` and `audit-ai-matrix.mjs` to read cloud tools from `cloud/components.tsx`.
- **Removed `tool-shim.ts` identity function:** eliminated no-op `extendedTool()` wrapper from 31 files (~38 import lines removed). Added `: any` type annotations to execute callback parameters to preserve strict-mode compatibility.
- **Created `tool-builders/shared-schemas.ts`:** reusable Zod schema fragments (`DATASET_NAME_PARAM`, `FIELD_NAME_PARAM`, `SHOW_ON_MAP_PARAM`, etc.) for tool builder consistency.
- **Decomposed `QMapToolContext` into 19 sub-interfaces:** new `context/types.ts` groups 200 properties by domain (DatasetOps, NumericOps, GeometryOps, ColorOps, MergeOps, UpsertOps, ToolRuntimeHelpers, FieldClassification, McpCloud, ZodSchemas, ThirdParty, WorkerOps, H3PaintOps, ToolComponents, KeplerActions, ConfigConstants). Composed via `extends` — structurally identical, zero consumer changes needed.
- **Split 5 large tool builder files (7278 LOC) into 16 focused modules:** `analytics.ts` -> 3 files (chart/statistical/timeseries), `dataset-derived.ts` -> 5 files (computed-field/delta/normalized/area/crs-reproject), `spatial-statistics.ts` -> 3 files (autocorrelation/hotspot/bivariate), `geometry-materialization.ts` -> 2 files (clip-by-geometry/zonal-stats), `tessellation-population.ts` -> 3 files (tessellation/h3-aggregation/population). Updated `use-tool-registry.ts` imports.
- **Added barrel files for qmap-ai subdirectories:** `middleware/index.ts`, `services/index.ts`, `context/index.ts`, `utils/index.ts` — re-export public API for cleaner imports.

### Added
- **Auto-retry transient eval failures:** `run-ai-eval.mjs` now identifies transient failures after the main loop (empty provider responses, transport errors, marginal precision misses) and retries them once. If the retry passes, the result is patched in-place so the report stays clean. Max 5 retryable cases per run (above that it's treated as systemic). Controlled by `QMAP_AI_EVAL_TRANSIENT_RETRY_MAX` (default `1`). Set to `0` to disable.
- **Unit tests for qmap-ai core modules (134 tests):** new `tests/unit/qmap-ai/` with test coverage for cache (AsyncMutex, bounded collections, serialization), guardrails (concurrency classification, phase gating), tool-contract (lookup, validation), tool-result-normalization (envelope wrapping, dedup eligibility), tool-args-normalization (Zod preprocessing, filter recovery).
- **Architecture documentation:** `docs/qmap-ai-architecture.md` — overview, mermaid diagram, directory structure, tool lifecycle, how-to-add-a-tool checklist, QMapToolContext sub-interface guide, cloud tools routing chain.

### Changed
- **Simplified Docker Compose dev stack:** removed `edge-only.yaml` from default `make up` in backends, so direct ports (3001/3004/3005) and Kong (:8000) are always exposed in dev. `backend-ready` is now a pure health-checker (no container start); start backends with `make dev-local`. Removed orphan targets (`up-ui`, `up-gateway`, `down-gateway`, `ps-gateway`, `logs-gateway`) and deleted `docker-compose.ui.yaml` (use `make same-origin-up` for prod-like UI+backend stack). `edge-only.yaml` remains used only by `same-origin-up` for deployment topology.

### Fixed
- **Contract validation used `qmapToolResult` envelope instead of `llmResult` for payload checks:** `_extract_request_tool_results` set `contract_validation_payload` to the thin `qmapToolResult` (success/details/schema only), skipping `llmResult` which carries tool-specific fields (`providers`, `datasets`, `providerId`, `datasetId`). This caused false `contractResponseMismatch` on every q-cumber discovery tool. Fixed by merging both objects for contract validation.
- **`objective_requests_map_display()` required "su mappa" suffix:** standalone display verbs like "mostra aree contaminate …" were not recognized as map display intent because the regex required "mostra … su mappa". Added leading-verb patterns (`mostra/visualizza/fai vedere/show/display`) at sentence start.
- **Display-intent `showOnMap=true` guardrail:** when `objective_requests_map_display()` detects a display objective, runtime now injects guidance on every sub-request requiring `showOnMap=true` for the final data query. Previously the model could freely choose `showOnMap=false` even for the final output, leaving the user with no visible layer.
- **Loaded-ref → catalog ID recovery for q-cumber queries:** `resolveQCumberDatasetId()` now detects when a loaded dataset reference (`*-query-qcumber-query-<ts>-<hash>`) is passed as `datasetId`, extracts the embedded catalog ID via suffix matching, and falls back to single-dataset auto-selection when the provider has only one dataset. Prevents cascade failures when the model confuses frontend loaded-refs with backend catalog IDs.
- **q-cumber queries with `loadToMap=true` did not trigger post-create validation:** `queryQCumberTerritorialUnits`, `queryQCumberDatasetSpatial`, `queryQCumberDataset` were not in `_DATASET_CREATE_OR_UPDATE_TOOLS`, so `hasDatasetMutation` stayed false and `waitForQMapDataset → countQMapRows` chain never fired. Added conditional check for `loadedToMap=true` in both backend quality metrics and frontend post-validation.
- **Garbled `<ctrlNN>` literal strings in response text:** some upstream providers (Gemini via OpenRouter) emit literal `<ctrl46>` escape sequences instead of actual control bytes. Added regex sanitization in `message_text.py` and stream response accumulation.
- **Dual/clip dataset tools reject model-hallucinated parameter aliases:** models (Gemini) systematically use wrong parameter names for multi-dataset tools (`sourceDatasetName`/`targetDatasetName` instead of `leftDatasetName`/`rightDatasetName`; `datasetName`/`maskDatasetName` instead of `sourceDatasetName`/`clipDatasetName`). Added Zod `z.preprocess()` normalizers: `preprocessDualDatasetArgs` for `spatialJoinByPredicate`, `coverageQualityReport`, `joinQMapDatasetsOnH3`; `preprocessClipDatasetArgs` for `clipQMapDatasetByGeometry`; `preprocessBoundaryClipArgs` for `clipDatasetByBoundary`.
- **ai-eval mock missing `providerId` in response:** `listQCumberDatasets` and `getQCumberDatasetHelp` mock results in `run-ai-eval.mjs` did not include `providerId`, causing contract response mismatch violations (`contractResponseMismatchRate=0.224`) on every functional eval run. Added `providerId` to all mock branches for both tools.
- **ai-cost-audit estimated prompt token thresholds:** raised `maxAvgEstimatedPromptTokens` (9k→22k), `maxP95EstimatedPromptTokens` (18k→33k), `maxMaxEstimatedPromptTokens` (22k→36k) to match current system prompt size growth. Old thresholds were set before tool contract schemas and runtime guardrail hints were added to the prompt.
- **ai-eval case `arch_existing_h3_aggregation_reuse_coloring`:** added `describeQMapField` to `expected_tools_any` — the model legitimately inspects field metadata before coloring H3 aggregation layers, causing a false `extraToolCalls=1` gate failure.
- **ai-eval retry on empty provider response:** `shouldRetryChatCompletion` now detects 200 OK responses with no content and no tool calls (provider returning empty completion) and triggers a retry. Previously these scored 0 and caused false quality-gate failures.
- **ai-eval-variance `avgExtraToolCalls` span threshold:** raised from 0.05 to 0.08 — observed LLM non-determinism on discovery tool calls produces natural span up to ~0.06 across runs.
- **Reverted incomplete help coverage guardrail:** `build_post_help_incomplete_coverage_decision` was added then removed in the same session. Forcing `getQCumberDatasetHelp` for ALL discovered datasets (up to 6) wasted the model's tool-call budget and caused it to use `queryQCumberDataset` instead of `queryQCumberTerritorialUnits`, regressing 5 data_pipeline cases from score 1.0 to 0.15-0.35. The correct fix for missing dataset metadata is the `argsSchema` in tool contracts (schema-first approach) + the existing `post_discovery_force_query` guardrail.
- **Filter `fieldName` → `field` normalization:** `normalizeQcumberFilterInput` now accepts `fieldName` as alias for `field` in q-cumber query filter objects. Models (Gemini) often confuse the two since other tools use `fieldName`. Fixes `AI_InvalidToolArgumentsError: Unrecognized key(s) in object: 'fieldName'` on `queryQCumberTerritorialUnits` filters.
- **Filter item schema in tool contracts:** q-cumber query tool contracts (`queryQCumberDataset`, `queryQCumberTerritorialUnits`, `queryQCumberDatasetSpatial`) now include explicit filter item schema with `field` as required property. Previously `filters` was bare `{type: 'array'}` with no guidance on filter object structure.
- **argsSchema for geometry/analysis tools in contracts:** added `STRICT_TOOL_ARGS_OVERRIDES` for `clipQMapDatasetByGeometry`, `zonalStatsByAdmin`, `spatialJoinByPredicate`, `overlayDifference`, `bufferAndSummarize` in `generate-tool-contracts.mjs`. Previously these tools had only `inputKeys` (flat name list) without type constraints or `required` fields, causing the LLM to hallucinate wrong parameter names. Now the contract exposes full JSON Schema with required fields, types, and enum constraints.
- **Pathological batch overflow fast-path:** when a model emits >2x `TOOL_CALL_RESPONSE_BATCH_CAP` tool calls in a single response (e.g. 49+ `waitForQMapDataset`), calls beyond 2x cap now return a frozen singleton `BATCH_OVERFLOW_RESULT` immediately — zero object allocation, zero events, zero state updates. Previously each overflow call created a new result object, triggered events, and ran `finalizeResult`, causing 49+ serial micro-operations that accumulated to timeout/AbortError.
- **Batch-skipped tool calls excluded from failure counts:** `computeAssistantExecutionStats` now distinguishes batch-skipped results (containing "skipped" + "batch" in details) from real failures. Batch-skipped calls are tracked separately as `skipped` and do not inflate `failed` count or trigger `completion_claim_blocked`/`non_success_outcome_annotation` guardrails. Progress line `[progress]` also excludes batch-skipped from `failed=N`. Fixes false `partial` status when Gemini emits batch tool calls that get legitimately capped.
- **Schema-first approach for geometry tool args:** removed `z.preprocess` dataset-arg normalization from all geometry tool builders. Per best practices (OpenAI strict mode, Anthropic tool-use-examples, Vercel AI SDK `.describe()`), wrong argument names should produce validation errors that flow back to the model for self-correction, not be silently normalized. The correct fix is `argsSchema` with `required` fields in tool contracts (added in this release) + clear `.describe()` annotations.
- **Circuit breaker override for discovery tools:** `getQCumberDatasetHelp` (max 8) and `listQCumberDatasets` (max 6) now have per-tool circuit breaker overrides in `tool-pipeline.ts`. Default cap of 3 was blocking the backend's `post_help_incomplete_coverage` guardrail when more than 3 datasets exist across providers.
- **Tool contract script paths:** fixed `POST_VALIDATION_PATH` in `generate-tool-contracts.mjs` and `audit-tool-contracts.mjs` to point to `services/post-validation.ts` (was `post-validation.ts`).
- **AI eval case expected tools:** added legitimate discovery/validation/domain tools (`listQCumberProviders`, `getQCumberDatasetHelp`, `checkRegulatoryCompliance`, `listQMapDatasets`, `queryQCumberDataset`, `waitForQMapDataset`, `countQMapRows`, `listQCumberDatasets`, `fitQMapToDataset`) to `expected_tools_any` for 7 functional eval cases that were incorrectly counting them as extra tool calls, inflating `avgExtraToolCalls` variance.
- **q-cumber query tools missing required provider/dataset:** `queryQCumberDataset`, `queryQCumberTerritorialUnits`, and `queryQCumberDatasetSpatial` declared `providerId` and `datasetId` as optional in the tool contract schema (`required: []`). The LLM omitted them on first call, producing `/providers//datasets` 404 errors on every request (visible as `failed_steps=1` in chat-audit). Added `['providerId', 'datasetId']` to `required` in the generator script and both generated contract JSON files.
- **PostGIS incremental dump restore:** replaced global `TABLE_COUNT > 1` guard in `010_restore_dumps.sh` with per-dump marker files (`PGDATA/.dump-markers/<file>.done`). New dumps added to `q-cumber-postgis/dumps/` are now automatically restored on next container start via `entrypoint-wrapper.sh`, not only on first DB initialization. Fixes OPAS tables (`opas_stations`, `opas_measurements`, `opas_hourly`) not being loaded when the dump was added after initial PostGIS bootstrap.
- **q-cumber proxy help path:** `qcumber_client.py` `get_dataset_help()` now calls `/providers/{id}/datasets/{did}/help` instead of `/providers/{id}/datasets/{did}`. Fixes 502/404 on `getQCumberDatasetHelp` when routed through q-assistant proxy.
- **Zero-match hallucination guardrail:** new `build_zero_match_must_acknowledge_decision` rule in `runtime_loop_limit_rules.py` detects when ALL filtered q-cumber queries returned 0 rows (from `details` "zero matches" indicator) and forces finalization with `tool_choice=none` + guidance to acknowledge no data found. Prevents the model from hallucinating row counts and station names when the requested data doesn't exist in PostGIS.
- **Ordinal/categorical coloring support:** `setQMapLayerColorByField` now accepts `mode: "ordinal"` (aliases: `categorical`, `category`, `unique`) for string/category fields like CLC `code_18`. Skips numeric validation, uses Kepler `ordinal` color scale with `viridis` default palette. Fixes `AI_InvalidToolArgumentsError` when the model correctly identifies a field as categorical.

### Added
- **Statistical analysis tools (simple-statistics):** 3 new AI tools backed by `simple-statistics` (9 kB gzip, zero deps):
  - `regressQMapFields` — linear regression between two numeric fields (slope, intercept, R², equation); optional derived dataset with `predicted` and `residual` columns
  - `classifyQMapFieldBreaks` — Ckmeans natural-break classification (Jenks-like) with break points and class counts; optional derived dataset with classification field
  - `correlateQMapFields` — pairwise Pearson correlation matrix between 2-10 fields with strength classification (strong/moderate/weak/negligible)
- **Regulatory compliance tools (D.Lgs. 155/2010):** 2 new AI tools backed by preloaded Italian air quality thresholds:
  - `listRegulatoryThresholds` — list limits for PM10/PM2.5/NO2/O3/SO2/CO/Benzene with optional EU 2030 targets (read-only, no dataset needed)
  - `checkRegulatoryCompliance` — per-station compliance check: exceedance counts, compliance rate %, applicable limit references, sorted by non-compliant stations first; supports `includeWho=true` (WHO AQG 2021) and `includeEu2030=true`
- **Population exposure assessment tool:** `assessPopulationExposure` joins measurement stations with nearby admin boundaries (haversine buffer, configurable radius 1-50 km), aggregates exposed population per station, and optionally checks regulatory compliance. Creates derived dataset with exposure columns when `showOnMap=true`.
- **IDW spatial interpolation tool:** `interpolateIDW` performs Inverse Distance Weighting from sparse point measurements to an H3 hexagonal grid, creating a continuous surface estimate. Configurable power (1-5), max neighbors (3-50), search radius (1-500 km), and H3 resolution (3-8). Output auto-detected as H3 layer by Kepler.
- **Expanded regulatory thresholds (v2):** JSON config now covers 12 pollutants (added Pb, As, Cd, Ni, BaP from Directive 2004/107/CE), WHO AQG 2021 guidelines, critical levels for vegetation (NO2=30, SO2=20, O3 AOT40=18000 µg/m³·h), PM2.5 exposure obligation (20 µg/m³), and O3 long-term objective
- **Superlative map-focus force-fit guardrail:** new `build_post_filter_force_fit_decision` in sequential chain forces `fitQMapToDataset` and strips all other tools after superlative winner is isolated and validated (rank→filter→wait→count). Addresses provider non-determinism (e.g. OpenRouter/Gemini ignoring `tool_choice`) that caused `fitQMapToDataset` to be skipped in 1/3 eval runs for `arch_superlative_admin_unit_show_on_map`. Dual enforcement with existing candidate scoring rule (score=159).
- **setQMapTooltipFields dataset ref resolution:** tooltip tool now uses `resolveDatasetByName` (handles `id:` prefix, normalized tokens, lineage aliases) instead of inline label/id match. Fixes `Dataset not found` when the model passes `datasetRef` (e.g. `id:qmap_tassellation_...`) instead of the raw dataset name.
- **getQCumberDatasetHelp catalog fallback:** when the help endpoint is unavailable but routing metadata can be inferred from the provider catalog, the tool now returns `success: true` with a warning instead of `success: false`. Prevents spurious `dataset_not_found` quality-metric noise and `completion_claim_blocked` guardrails on subsequent steps.
- **H3 join low-coverage retry hint:** `joinQMapDatasetsOnH3` error message now suggests `minCoveragePct` override value when coverage is below threshold, enabling the model to retry with an explicit lower gate when the user confirms.
- **Fit preserved for mutation workflows:** `preserve_fit_without_explicit_map_focus` now activates when any dataset-creating tool (tessellation, aggregate, join, clip, query, etc.) succeeded in results. Previously fit was pruned unless the prompt contained explicit map-display keywords, forcing users to say "mostramelo" after every mutation. Covers 17 mutation tools via `has_mutation_workflow_signal` in `runtime_workflow_state.py`.
- **Map viewport initialization cleanup:** removed `registerEntry` pre-dispatch, `mint={false}`, `resolveQMapMountViewport` workaround, and 250ms timeout retry. These workarounds were causing the viewport flash to 0,0 (Africa) instead of preventing it. Now follows the standard kepler.gl pattern: `keplerGlReducer.initialState({mapState: {latitude: 42.5, longitude: 12.5, ...}})` handles initialization, no extra dispatches needed.
- **Spurious GET to api.openai.com on boot:** pre-dispatch `updateAiAssistantConfig` synchronously at store creation with the q-map proxy URL, replacing the kepler core default (`api.openai.com/v1`). Removed the redundant `useEffect`-based config dispatch.
- **Dataset/layer resolution with `id:` prefix:** 7 styling/visibility tools (`setQMapLayerColorByField`, `setQMapLayerHeightByField`, `setQMapLayerSolidColor`, `applyQMapStylePreset`, `setQMapLayerColorByThresholds`, `setQMapLayerColorByStatsThresholds`, `setQMapLayerVisibility`) now use `resolveDatasetByName` instead of inline label/id match. Fixes "layer/dataset not found" when the model passes `datasetRef` format (`id:...`).
- **Turn-level tool call hard cap:** tool pipeline now enforces a total of 15 tool calls per turn (all tools combined), in addition to the existing per-tool circuit breaker (max 3). Prevents providers that ignore `parallel_tool_calls=false` (e.g. Gemini/OpenRouter) from burning through 30+ batched calls before abort.
- **Circuit breaker counter never reset between user messages:** `extractToolPolicyUserText` only checked `payload.message` (singular, OpenAssistant format) but the SDK sends `payload.messages` (plural, OpenAI-compatible array). Counter accumulated across all user messages in the session. Now checks the last message in the array — resets on new user turn, persists across sub-requests.
- **Response batch truncation (cap=3):** tool pipeline enforces max 3 tool executions per LLM response. When the provider emits 4+ tool calls in a batch (ignoring `parallel_tool_calls=false`), calls beyond the 3rd return a skip message. Allows legitimate batches (showOnly+fit+tooltip) but blocks pathological ones (4+ wait, 16 count). Tracked via `responseBatchTracker` reset on each fetch response.
- **Cloud map routing:** `listQMapCloudMaps`/`loadCloudMapAndWait`/`loadQMapCloudMap` are now pruned at runtime unless the user explicitly requests cloud/saved/personal maps. Adds `[MUST]` prompt rule (EN+IT) and `build_cloud_tools_require_explicit_request_decision` runtime guardrail with `_objective_mentions_cloud_or_saved_maps` intent detector. Prevents territorial/analytical queries from loading saved maps with stale H3 layers and non-zero pitch.
- **q-cumber proxy auth forwarding:** `qcumber_proxy.py` and `qcumber_client.py` now forward the caller's `Authorization` bearer token to q-cumber instead of using an empty static `Q_ASSISTANT_QCUMBER_CLOUD_TOKEN`. Fixes 401 Unauthorized on all q-cumber requests routed through q-assistant in the platform-local stack.
- **Stateless dedup cache collision:** `queryQCumberTerritorialUnits`, `queryQCumberDataset`, and `queryQCumberDatasetSpatial` removed from `STATELESS_TOOL_DEDUP_ELIGIBLE_TOOLS` because `loadToMap=true` mutates map state; the dedup cache was returning stale results from a previous query (e.g. regions instead of provinces), preventing the new dataset from being created.
- **Full tool serialization:** ALL tool calls now go through the `AsyncMutex` (not just mutations), so they execute one at a time in FIFO order. This prevents the LLM from batch-planning tool chains where downstream tools reference dataset IDs that don't exist yet. `parallel_tool_calls=false` is also set in the backend payload, but frontend serialization is the reliable enforcement since not all providers (e.g. Gemini/OpenRouter) support it.
- **Sequential tool calls enforced:** `parallel_tool_calls: false` injected in `_coerce_openai_chat_payload` when tools are present. Forces the model to emit one tool call per response, preventing batch-planned chains with stale/guessed dataset IDs. This is the root fix for cascading failures in multi-step workflows.
- **Per-tool circuit breaker:** max 3 calls per tool name per user message; blocks execution with structured error after the cap, preventing infinite local-only tool loops (e.g. 168 `countQMapRows` observed in production). Counter persists across sub-requests and resets only on new user message. `countQMapRows` also added to `STATELESS_TOOL_DEDUP_ELIGIBLE_TOOLS` for intra-turn dedup.
- **Post-discovery forced query transition:** new `build_post_discovery_force_query_decision` runtime guardrail forces the model to proceed with `getQCumberDatasetHelp` or `queryQCumberTerritorialUnits` after discovery completes (listProviders + listDatasets both succeeded, no query issued). Implements the deterministic state-machine transition `discovery → query` via `tool_choice`, preventing models (e.g. Gemini) from stopping after discovery without querying data.
- **System prompt sequential tool chaining:** added `[MUST]` rule (EN+IT) instructing the model to not emit the entire tool chain in a single response, since output dataset IDs are not known until each creation step completes.
- **System prompt canonical dataset ref:** added `[MUST]` rule (EN+IT) requiring the model to use `loadedDatasetRef` from `queryQCumberTerritorialUnits` results as input for downstream tools, instead of reusing stale refs from previous queries/turns.
- **System prompt layer isolation:** added `[MUST]` rule (EN+IT) requiring `showOnlyQMapLayer` on the final filtered/derived layer after wait/count validation, to hide source and intermediate layers from previous steps.
